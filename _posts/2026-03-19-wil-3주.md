<h2><b>🎯 목표 설정 및 수행 결과 회고</b></h2>
<p><b>1. 문제 해결 (Algorithm)</b></p>
<ul>
<li><b>목표:</b> 알고리즘 카테고리에 따라 매일 1문제 스스로 풀기</li>
<li><b>결과:</b> 매일 실천에 어려움이 있었으며, 블로그 참조와 AI 코드 리뷰에 의존함</li>
<li><b>Action Plan:</b> 30분간은 외부 도움 없이 스스로 풀고, 이후 즉시 피드백(블로그, AI)을 받는 방식으로 전환</li>
</ul>
<p><b>2. 설계 (Architecture)</b></p>
<ul>
<li><b>목표:</b> 확장 가능한 폴더 구조 설계로 협업 시 Git 충돌 방지</li>
<li><b>결과:</b> Redis 구현 시 테스트 폴더 구조를 분리하여 바이브 코딩 중 코드 혼선 방지 성공</li>
</ul>
<p><b>3. 구현 (Implementation)</b></p>
<ul>
<li><b>목표:</b> 바이브 코딩 시 명세서를 작성하여 AI의 프로젝트 문맥 파악 돕기</li>
<li><b>결과:</b> 명세서 기반 학습은 이루어졌으나, Google Docs 사용으로 가독성 저하</li>
<li><b>Action Plan:</b> 향후 Notion 활용 및 명세서 자동화 방안 모색</li>
</ul>
<p><b>4. 품질 (Quality)</b></p>
<ul>
<li><b>목표:</b> Redis 관련 프로젝트의 모든 테스트 통과</li>
<li><b>결과:</b> Github Actions로 자동화 구축. Main 브랜치 수정 시 테스트 자동 실행 및 결과를 Notion에 공유하는 시스템 마련</li>
</ul>
<p><b>5. 유지보수 (Maintenance)</b></p>
<ul>
<li><b>목표:</b> 주석 작성을 통한 팀원 간 이해도 증진</li>
<li><b>결과:</b> 기능 구현에 집중하느라 주석 및 유지보수 고려 부족 (미흡)</li>
</ul>
<p><b>6. 협업 (Collaboration)</b></p>
<ul>
<li><b>목표:</b> 능동적인 태도로 협업 속도 향상</li>
<li><b>결과:</b> 명세서 사전 작성을 통해 팀원들의 프로젝트 이해도 및 작업 속도 증진</li>
</ul>
<p><b>7. 태도 (Mindset)</b></p>
<ul>
<li><b>목표:</b> Redis의 코어 개념과 원리를 학습 후 구현</li>
<li><b>결과:</b> 캐시 개념은 공부했으나 실무 적용에 한계. '왜 우리에게 Redis가 필요한가'에 대한 논리적 근거 부족</li>
</ul>
<p><b>8. 비즈니스 이해 (Business Insight)</b></p>
<ul>
<li><b>목표:</b> 사용자 편의성(UX)을 고려한 코드 및 프로젝트 진행</li>
<li><b>결과:</b> 사용자 관점보다는 기술적 구현에만 매몰됨 (미흡)</li>
</ul>
<p><b>9. AI 활용 (AI Prompting)</b></p>
<ul>
<li><b>목표:</b> Agent.md 최적화를 통한 고품질 코드 생성</li>
<li><b>결과:</b> OpenAI 팁을 참고하여 Agent.md를 영문으로 작성하고 한국어 설명을 병기하여 효과적인 가이드 마련</li>
</ul>
<p><b>10. 학습 민첩성 (Learning Agility)</b></p>
<ul>
<li><b>목표:</b> Redis 내부 구조 이해 및 구현</li>
<li><b>결과:</b> 깊이 있는 이해 부족으로 Python 기본 딕셔너리를 무분별하게 사용</li>
<li><b>Action Plan:</b> 내부 구현 사항을 선행 학습하고, 기술 선택 시 구체적인 이유를 제시할 수 있도록 준비</li>
</ul>

<h2><b>🔍 4주차 발제: 더 깊게 고민할 사항</b></h2>
<h3><b>1. 파이썬 자료구조의 내부 구현</b></h3>
<ul>
<li><b>List:</b> 연결 리스트(Linked List)가 아닌 동적 배열(Dynamic Array)로 구현됨</li>
<li><b>Dictionary:</b> Hash 방식 사용. Python 3.6+부터 메모리 효율을 위해 <b>인덱스 테이블</b>과 <b>엔트리 테이블</b>을 분리
<ul>
<li>데이터 삽입 순서가 유지되도록 개선됨 (Open Addressing 기반)</li>
</ul>
</li>
<li><b>Sort():</b> <b>Timsort</b> (Insertion + Merge Sort 혼합) 사용
<ul>
<li>최선: O(n), 최악: O(n \log n)의 성능 보장</li>
</ul>
</li>
</ul>
<h3><b>2. WebAssembly (Wasm)와 저수준 언어의 필요성</b></h3>
<ul>
<li><b>현상:</b> JS는 인터프리터 기반이라 복잡한 연산(영상 편집, 3D, 암호화)에서 병목 발생</li>
<li><b>해결:</b> C/Rust로 로직 설계 후 Wasm으로 컴파일하여 네이티브에 가까운 속도 구현</li>
<li><b>전략:</b> Python으로 빠르게 프로토타입 제작 후, 성능이 필요한 구간은 빠른 언어로 전환</li>
</ul>
<h3><b>3. Redis Hash 직접 구현 시 고려사항</b></h3>
<ul>
<li>타 언어에서 직접 Hash를 구현해야 한다면 다음 세 가지가 핵심임:
<ol>
<li><b>해시 함수 선정</b></li>
<li><b>충돌(Collision) 처리 방식</b> 결정</li>
<li><b>동적 리사이징:</b> 데이터 임계치 초과 시 테이블 크기를 확장(보통 2배)하고 재배치(Rehashing)</li>
</ol>
<b>Note:</b> Redis는 소량 데이터일 땐 ziplist(압축 구조), 많아지면 hashtable로 전환하는 유연성을 가짐</li>
</ul>
<h3><b>4. 바이브 코딩 결과물에 대한 '진짜' 내 것 만들기</b></h3>
<ul>
<li>AI가 만든 결과물에 대해 답변하지 못한다면 그것은 본인의 실력이 아님</li>
<li><b>수요코딩회 활용:</b> 내가 완벽히 소화할 수 있는 범위만큼만 AI를 활용하고, 그 내용을 공유하며 검증받기</li>
</ul>
